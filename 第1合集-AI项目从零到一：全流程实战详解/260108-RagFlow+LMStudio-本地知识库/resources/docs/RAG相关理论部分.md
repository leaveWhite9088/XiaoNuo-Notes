#### 嵌入模型

- 嵌入的目的，是进行语义相似度的对比，进一步地，捕捉词语之间语义和句法的关系
- 词嵌入 & 上下文嵌入
  - 静态词嵌入模型（Glove、Word2Vec）：对于同一个词，在不同情况下，词的嵌入的结果相同
  - 动态上下文嵌入模型：对于同一个词，得益于自注意力机制，在不同上下文下，生成的结果不同

- embedding一般针对token，为什么在rag索引的时候，一整段话也可以embedding？
  - 现代的句子嵌入模型（如 Sentence-BERT、OpenAI 的 text-embedding-ada-002）经过专门训练，能够将**变长的文本序列（如一个句子、一个段落甚至更长的文本）** 映射为一个固定维度的稠密向量。这个过程通常通过对模型最后一层隐藏状态的所有 token 向量进行池化（Pooling）操作（如均值池化）来实现，从而捕获整段文字的全局语义信息

现代句子嵌入模型（如SBERT和OpenAI的嵌入模型）大多基于**Transformer编码器架构**。其工作流程可以概括为以下几步：

1. **文本预处理与分词**：将输入文本分割成模型能够理解的 token（通常是词或子词），并添加必要的特殊标记（如`[CLS]`, `[SEP]`）。
2. **生成Token嵌入**：模型将每个 token 转换为一个高维向量表示。在Transformer的每一层中，**自注意力机制（Self-Attention）** 都发挥着至关重要的作用。
3. **上下文编码**：经过多层Transformer块的处理，每个token的向量都**融合了整个输入序列中所有其他token的信息**，形成了丰富的上下文表示。
4. **池化（Pooling）操作**：这是将**变长序列转换为固定维度向量**的关键一步。模型最后一层所有token的隐藏状态需要通过池化操作来聚合信息。常见策略有
   - **均值池化（Mean Pooling）**：计算所有token向量的平均值。这是最常用且有效的方法之一，能稳健地捕捉整体语义。
   - **`[CLS]`标记池化**：BERT等模型在输入开头会添加一个特殊的`[CLS]`标记。训练过程中，该标记的最终隐藏状态被设计为承载整个序列的概要信息。一些模型直接使用这个向量作为句子表示。
   - **最大池化（Max Pooling）**：取所有token向量在每个维度上的最大值，试图捕捉最显著的特征。
   - **加权平均池化**：根据token的重要性（如TF-IDF权重或模型学到的权重）进行加权平均。
5. **输出固定维度的句子向量**：池化后得到一个固定维度的稠密向量，作为整个输入文本的语义表示。

#### 向量数据库

- 不同的向量数据库支持不同的索引种类和数量、并发、拓展等等
  - **数据规模与性能要求**：需要处理**十亿级以上**的向量数据并追求**毫秒级检索**，**Milvus**或 **Qdrant**可能更合适。若数据量在**百万级以下**或主要用于**原型开发**，**Chroma**或 **Pgvector**更轻便。
  - **部署与运维能力**：若团队拥有较强的运维技术实力，倾向于**私有化部署**和高度定制，开源方案如 **Milvus**、**Qdrant**是选择。若希望**最小化运维投入**，快速开始，**全托管的云服务**如 **Pinecone**更省心。
  - **功能与生态集成**：需要进行**混合查询**（结合向量相似性和结构化条件过滤），**Weaviate**或 **Pgvector**表现不错。技术栈若深度绑定 **PostgreSQL**，**Pgvector**能无缝集成。若与 **LangChain**、**LlamaIndex** 等大模型框架集成是重点，**Pinecone**、**Chroma**及 **Weaviate**通常有良好支持。
  - **成本考量**：**开源方案**前期硬件和运维成本可能较高。**云托管服务**按用量付费，需注意随数据量和查询量增长可能带来的成本上升。
  - **特殊需求**：对**国产化信创**有要求，可考虑 **腾讯云 VectorDB**。需要进行**多模态检索**（如图文跨模态），**Weaviate**或 **Vespa**支持较好。

#### 重排片段

- 确保最相关的信息被优先考虑
- 一般采用一个“跨编码器（Cross encoder model）”，将检索到的块和查询一起评估，计算相关性分数

![1758634227614](.\resources\1758634227614.png)

- Cross Encoder（交叉编码器）是一种用于处理文本对（如查询和文档）任务的深度学习模型架构。它通过联合编码和深度语义交互，能够非常精确地判断两个文本片段之间的相关性，常被用于**检索增强生成（RAG）系统**中的精排（Rerank）阶段、问答匹配、自然语言推理（NLI)等任务**Cross Encoder 如何工作**：

- Cross Encoder 的核心在于其**联合编码**和**深度交互**的机制

  - **输入表示**：Cross Encoder 会将查询（Query）和候选文档（Document）**拼接成一个序列**作为输入。例如，在 BERT 风格的模型中，格式通常为：`[CLS] Query [SEP] Document [SEP]`

  - **联合编码与深度交互**：这个拼接后的序列会被送入 **Transformer 编码器**（如 BERT、RoBERTa、DeBERTa 等）。Transformer 的**自注意力机制（Self-Attention）** 允许模型在编码过程中，让查询中的每个词（Token）与文档中的每个词进行交互，反之亦然。这种**词级别的细粒度交互**能力，是 Cross Encoder 能够精准理解语义关系的关键

  - **输出与评分**：模型最终取 `[CLS]`标记对应的输出向量作为整个序列的表示，并通过一个简单的线性分类层（Linear Head）输出一个表示相关性的分数（如通过 Sigmoid 函数变换为 0 到 1 之间的值）或分类结果

| 特性维度         | Cross Encoder (交叉编码器)                              | Bi-Encoder (双编码器)                                        |
| :--------------- | :------------------------------------------------------ | :----------------------------------------------------------- |
| **输入处理方式** | 将查询（Query）和文档（Document）**拼接**后一起输入模型 | 查询和文档**分别独立编码**，生成各自的向量表示               |
| **语义交互时机** | **编码过程中**进行深度的、细粒度的语义交互              | 编码过程**无交互**，依赖编码后向量的相似度计算               |
| **输出形式**     | 直接输出一个**相关性分数**或分类结果                    | 输出**向量**，通过计算向量间的余弦相似度或点积来衡量相关性   |
| **计算效率**     | **较低**，每次计算需对查询-文档对进行完整的前向传播     | **很高**，文档向量可预先计算和索引，检索时仅需计算查询向量和相似度搜索 |
| **精度表现**     | **高**，能捕捉复杂的语义关系和细微差别                  | **相对较低**，因信息压缩和缺乏即时交互，可能丢失细节         |
| **预计算支持**   | ❌ 不支持，必须实时计算                                  | ✅ 支持，文档向量可提前计算并存入向量数据库                   |
| **典型应用场景** | 精排（Rerank）、问答系统、高精度语义匹配任务            | 大规模检索的初筛（粗排）、语义相似性计算、需要快速响应的场景 |

## RAG分块（chunking）策略

### 为什么需要分块

- Embedding长度限制

  - 比如 OpenAI 的 text-embedding-3-small，Hugging Face 的 all-MiniLM-L6-v2对输入文本又严格限制

  - 如果不分块，文档太长就无法进行embedding

- 提高语义相关性和精度

  - 如果每一块太大，就会混杂多个主题，导致匹配不准确；还会导致某个块仅仅部分内容相关，却被整个大块拖累
  - 如果分块恰当，就可以使得语义集中、匹配度更高；检索出来的块的相关性也会更高

- 避免上下文污染、降低生成负担

  - 大文档直接给大模型会导致成本高、速度慢；降低模型对真正相关信息的关注度；增加模型幻觉风险
  - 适当的分块+Top-K的检索，可以确保只把最相关的内容送入模型

- 支持增量更新

  - 分块后的文档更适合做局部知识更新

### RAG分块策略

#### 固定大小分块

![1758619250260](.\resources\1758619250260.png)

- 优点：容易实现
- 缺点：会造成语句在中间被分割，导致重要信息在不同的块中

#### 语义分块

![1758619351902](.\resources\1758619351902.png)

1. 把文档分成句子或者段落
2. 对每一个分段进行embedding
3. 如果第一个向量化分段与第二个向量化分段具有较高的相似度（余弦相似度），则将两端合并为一个分段
4. 然后将得到的这个新分段与原本的第三个分段进行相似度对比，一直持续到相似度显著下降
5. 一旦下降，就将这个得到的新分段作为一个分块
6. 重复步骤3-5，直到最后一个分段判断完毕

![1758619615972](.\resources\1758619615972.png)

- 优点：保持语句连贯性、语义连贯性，可以明显提高检索准确度，使得LLM生成更连贯的相应
- 依赖于阈值来确定余弦相似度是否显著下降。对于不同的文档，可能这个阈值不一样，所以不太好实施

#### 递归分块

![1758619766744](.\resources\1758619766744.png)

1. 基于段落、章节等固有的分隔符进行分块
   1. 往往是基于段落来的，不一定要有非常明确的文档结构
2. 然后，如果分块的大小超过限制，就把这个块拆分为两个小块。

![1758619899236](.\resources\1758619899236.png)

- 优点：和固定大小分块来比的话，保证了语义语句的连贯性；和语义分块来比，实现复杂度有所降低
- 缺点：实现的时候，会有一些额外的计算开销

#### 文档结构分块

![1758620042639](.\resources\1758620042639.png)

- 就是利用文档的固有结构，比如标题、章节或者段落，来定义信息块的边界。
- 通过与文档的逻辑章节保持一致来维持结构的完整性

![1758620157994](.\resources\1758620157994.png)

- 优点：结构清晰，分块简单
- 缺点：过于理想化，文档未必有清晰的结构。片段长度不同，也可能会超出模型上下文限制
  - 解决办法：结合递归分块

#### 基于LLM的分块

![1758620314269](.\resources\1758620314269.png)

- 这种方法非常简单粗暴，直接把文档丢给大模型，让LLM根据设定的提示词生成语义上隔离且具有意义的片段。

- 优点：实现最容易
- 缺点：计算需求最高

## RAG 评估

### RAG系统的准确性体现

- 基准测试好，不一定垂直领域关键细节好
  - 对外行来说的完全相同的查询，在专家看来，可能有很大差别。通用嵌入模型可能会忽略这些细微差别
  - 如果Rerank的模型没有经过专业术语、行业黑话、缩略词训练，可能会将相关信息误判为不相关
  - 大语言模型可能对特定领域下的专业称呼、缩写、或超出其训练数据范围外的知识产生认知混乱

### RAG评估的指标

![1758699741067](.\resources\1758699741067.png)

#### 检索器

- 准确率：系统检索出的实际相关文档数 / 系统检索出的文档数。用于评估结果准不准
- 召回率：系统检索出的相关文档 / 所有实际相关的文档。用于评估结果全不全

#### 生成器

- 忠实度：指生成结果是否基于检索的结果，事实准确。用于评估生成内容是否谎言
- 相关性：是指生成结果是否和检索内容是强相关的。用于评估生成内容是否答非所问
- 有时候会出现忠实度高而相关性低的情况
  - 比如说，回答把检索到的原文抄了下来，但是检索到的原文和问题不相关

### RAG检索评估

- 检索评估的目标就是评估根据查询从知识库检索结果整个过程是否成功。
- 注意我这里说的是“知识库”，而非向量库。也就是说，无论数据库、搜索引擎、文件系统、向量库、网络还是任何可以存储数据的地方，都属于知识库的范畴。RAG是检索增强生成，它并没有强制知识库一定是向量库。

#### 检索评估准召率定义

| 指标名称                      | 核心定义                                                     | 计算公式                                           | 核心关注点                                    | 典型应用场景                             |
| :---------------------------- | :----------------------------------------------------------- | :------------------------------------------------- | :-------------------------------------------- | :--------------------------------------- |
| **召回率 (Recall)**           | 系统成功检索到的相关文档占知识库中**所有相关文档**的比例。   | `召回率 = TP / (TP + FN)`                          | 检索的**全面性**，是否遗漏关键信息。          | 法律、医疗等要求信息完备的领域。         |
| **准确率/精确率 (Precision)** | 检索到的文档中，**真正相关**的比例。                         | `精确率 = TP / (TP + FP)`                          | 检索结果的**准确性**，是否包含过多噪声。      | 客服机器人等需要快速精准响应的场景。     |
| **命中率 (Hit Rate@K)**       | 在前K个检索结果中，**至少包含一个**相关文档的查询所占的比例。 | `Hit@K = (至少命中1个相关文档的查询数) / 总查询数` | 系统在前K个结果中**快速找到相关信息**的能力。 | 通用检索系统的初步效果评估。             |
| **MRR (平均倒数排名)**        | 对每个查询，**第一个相关文档排名倒数**的平均值。             | `MRR = (1/Q) * Σ(1 / rank_i)`                      | **第一个相关结果**出现的位置，强调检索速度。  | 搜索引擎，重视首条相关结果。             |
| **NDCG (归一化折扣累积增益)** | 考虑结果排序和文档**不同相关程度**的加权评分。               | `NDCG = DCG / IDCG`                                | 检索结果的**排序质量**和相关性等级。          | 电商推荐、搜索排序等对排序要求高的场景。 |

##### 指标详解与应用考量

- 召回率 (Recall) 与 精确率 (Precision)：这对指标常常需要权衡。追求高召回率（扩大检索范围）可能会引入不相关文档，降低精确率；而追求高精确率（缩小检索范围）则可能导致遗漏相关文档，降低召回率。
  - **平衡工具：F1分数**。它是精确率和召回率的调和平均数，用于综合评估系统性能，尤其在需要平衡两者时非常有用。公式为：`F1 = 2 * (Precision * Recall) / (Precision + Recall)`。
  - **@K变体**：在实际应用中，我们更关注前K个结果的质量，因此产生了 `Precision@K`和 `Recall@K`。例如，对于RAG系统，如果生成模块只使用前5个检索结果，那么 `Precision@5`就比整体精确率更重要。

- MRR (平均倒数排名)：MRR特别关注系统**定位第一个相关答案的效率**。它计算的是每个查询中第一个相关文档排名的倒数，再对所有查询求平均。

  - **例子**：假设有3个查询，其第一个相关文档的排名分别为2、3、1。则MRR = (1/3) * (1/2 + 1/3 + 1/1) ≈ 0.61。

  - **特点**：MRR值越高，说明用户能越快找到第一个有用结果。但它只关心第一个相关文档，忽略后续其他相关文档的质量。

- NDCG (归一化折扣累积增益)：NDCG是一个更为精细的指标，NDCG通过将实际检索结果的累积收益(DCG)与理想排序下的最大累积收益(IDCG)进行比较来评估排序质量，其值越接近1，说明排序效果越好。它同时考虑了两个方面：
  - **位置折扣**：排名越靠后的结果，其价值会打折扣，因为用户更关注前列结果。
  - **相关性等级**：文档的相关性不是简单的“相关”或“不相关”，而是可以有程度之分（例如，高度相关=2分，一般相关=1分，不相关=0分）。

##### 如何有效运用这些指标

1. **明确评估目标**：根据你的业务场景确定优先级。是要求信息尽可能全面（高召回率），还是要求答案精准简洁（高精确率）？
2. **使用@K指标**：结合你的RAG管道实际使用的上下文窗口大小（例如，Top-5或Top-10）来评估 `Recall@K`和 `Precision@K`，这样更具实际意义。
3. **综合评估**：不要孤立地看单个指标。可以结合使用MRR（关注首条结果）和NDCG（关注整体排序质量）来全面评价检索效果。
4. **借助评估框架**：可以尝试使用专业的RAG评估框架（如 **RAGAS**），它们通常内置了这些指标的自动化计算，能帮助你系统化地进行评估和优化。

- 第一步：构建黄金数据集
- 第二步：运行RAG系统。输入问题，返回前K个检索结果
- 第三步：评估检索结果。进行各种指标的计算。

#### 检索评估实践经验

| 问题 | 建议 |
| :-- |:-- |
| 检索不到正确信息 | 可能是embedding模型不匹配语料库（试试bge、qwen3-embedding、bce等）。知识含有大量缩写、行业黑化、专业术语的时候，embedding模型的捕捉效果差（如：HW可能表示华为、家庭作业、海外等），试试微调embedding模型。 |
| 正确文档排得太后 | 考虑加入rerank模型（Cohere Rerank，bge-reranker等） |

| 检索内容太长或者太短 | 合理切片文段（如每段200-500字），提升可检索性 |

##### RAG准召率的权衡

- 召回率的准确率通常存在此消彼长的关系
  - 为了提高召回率，也就是要找到更多的文档，通常需要更宽松的条件，比如**降低向量相似度的要求**、**扩大检索窗口**等，但这样做往往会检索出更多的不相关的文档，从而减低准确率
  - 为了提高准确率，也就是要返回最相关的文档，通常需要收紧条件，比如**提高向量相似度阈值**、**减少返回数量**等，但这样做往往会忽略掉一些相关但边缘的文档，从而减低召回率
- 理想情况是都要，但是必须要做出取舍
  - 当面对高完整性场景的时候（比如问答、事实核查），召回率更重要，要保证关键信息不遗漏。然后可以通过大语言模型的鲁棒性，去自动去除一些噪声，或者后续通过rerank来提高准确度
  - 当面对高准确性场景的时候（比如医疗、法律、代码补全），准确率就更重要，要保证LLM接收到的都是相关信息，防止被无关信息带偏

##### 实际运用中的策略（没有银弹）

###### 提升召回率

- 优化分块策略：重叠分块、按语义/句子分块、多粒度分块、避免关键信息切分在边界造成的信息丢失
  - **重叠分块**：**是什么**：相邻的两个文档块之间保留一部分重复的文本。**为什么有效**：解决“边界效应”问题。比如一个关键信息正好在某个 chunk 的末尾，另一个相关信息在下一个 chunk 的开头。如果不重叠，检索第一个 chunk 时，由于关键信息在末尾，其向量表征可能不完整，导致漏检。重叠确保了关键信息能完整地出现在至少一个 chunk 的中心位置。**如何做**：设置一个重叠长度（如 100-200 字符）。常见工具如 LangChain 的 `RecursiveCharacterTextSplitter`支持此功能。
  - **按语义/句子分块**：**是什么**：不以固定的字符数分块，而是根据句子、段落等自然语言边界进行划分。**为什么有效**：固定长度分块可能会把一个完整的句子或思想拦腰截断，破坏语义完整性。按句子分块能保证每个 chunk 在语义上更自洽，便于嵌入模型理解。**如何做**：使用句子分割器（如 NLTK, spaCy）先分句，再将以一定数量的句子组合成一个 chunk。
  - **多粒度分块**：**是什么**：对同一份文档采用多种不同大小的分块策略（如小 chunk：256字符，用于细节检索；大 chunk：1024字符，用于概述性检索）。**为什么有效**：问题本身有不同粒度。例如，“总结某某文章”需要大 chunk；“某某文章提到了哪些关键技术”需要更细粒度的小 chunk。检索时，可以同时检索两种粒度的 chunk，然后合并或选择。**如何做**：构建多套向量库，或使用如 `LlamaIndex`的父子索引结构，小 chunk 作为子节点，并链接到包含它的大 chunk（父节点）。
- 改进检索器：尝试不同的嵌入模型，或者进行微调，使更贴合下游任务；结合关键词检索（BM25）与向量检索，也就是混合检索
  - **尝试不同的嵌入模型**：**是什么**：不要只满足于 OpenAI 的 `text-embedding-ada-002`。开源社区和其他厂商有很多优秀的模型。**为什么有效**：不同的模型在不同领域和任务上表现各异。例如，`bge-large-zh`在中文任务上表现优异，`M3E`是针对中文优化的开源模型。**如何做**：在 MTEB 或 CMTEB 等评测榜上寻找适合你任务（特别是语种和领域）的模型进行替换测试。
  - **对嵌入模型进行微调**：**是什么**：使用你所在领域的专有数据对通用的嵌入模型进行继续训练。**为什么有效**：让模型学习你领域的专业术语、表达习惯和相似性标准。例如，在医疗领域，“感冒”和“流感”的相似度，与日常对话中的理解可能不同。**如何做**：需要准备一个由（查询，正例文档，负例文档）组成的三元组训练数据，使用对比学习进行微调。
  - **混合检索**：**是什么**：结合**向量检索**（语义相似度）和**关键词检索**（如 BM25）。**为什么有效**：两者优势互补。**向量检索**：擅长语义匹配，能解决“一词多义”和“多词一义”问题（例如搜索“苹果公司”，也能找到提到“iPhone”的文档）。**关键词检索**：擅长精确匹配，对于专有名词、代号、缩写等检索非常直接有效。**如何做**：分别进行两种检索，然后对结果进行融合。常见融合策略有：**加权融合**：给两个检索结果的分数赋予权重，计算综合分。**RRF**：一种更鲁棒的融合方式，不依赖于分数本身的大小，只关注排名位置。
- 扩大搜索范围：适当增加top-k（返回的文档的数量）、降低相似度阈值
- 查询拓展/改写：使得原始查询更清晰或者丰富
  - 让用户的原始查询变得更“好”，更容易被检索器理解。**是什么**：在检索前，先用 LLM 对用户查询进行加工。**为什么有效**：用户查询往往是简短、模糊或有语法错误的。**如何做**：**改写**：将口语化查询改写成更书面、完整的句子。`将“苹果最新手机有啥功能” -> “请介绍最新款 iPhone 的主要功能和特性”`**扩展**：根据查询，让 LLM 生成几个相关的、同义的问题。例如原问题“如何学习深度学习”，可以扩展出“深度学习入门指南”、“深度学习学习路径”、“深度学习资源推荐”等，然后用这些扩展后的问题一起去检索。

###### 提升准确率

- 重排序rerank（最有效）：从top-k到top-n
- 优化检索器：同样可以尝试不同的嵌入模型或者微调，使其区分相关/不相关的文档的能力更强
  - 目标从“找到更多相关文档”变为“**更好地区分相关与不相关文档**”。微调嵌入模型时，训练数据的负例要精心构造，包含那些“看似相关实则不相关”的难负例。
- 收紧检索范围：提高相似度阈值，适当减少top-k
- 元数据过滤：如果文档有更清晰的元数据（日期、来源、类型），可以利用其进行过滤
  - 如果文档自带高质量元数据，这是非常强大的工具。**是什么**：利用文档的元信息（如创建日期、作者、部门、标签、类型[是用户手册还是API文档]等）在检索前或检索后进行过滤。**为什么有效**：能极大缩小搜索范围，提升精准度。例如，用户问“我们公司最新的请假政策是什么”，可以过滤元数据 `文档类型=“公司政策”`并按 `发布日期`倒序排列。**如何做**：在向量数据库（如 Weaviate, Pinecone）中，可以将元数据作为筛选条件，在向量检索的同时进行过滤。
- 优化分块策略：保证分块内的信息有较好的语义完整性，避免包含过多的无用内容
  - **保证语义完整性**：避免一个 chunk 内包含多个不相关的主题，这会让检索器困惑。确保每个 chunk 围绕一个核心意思。**避免无用内容**：在分块前进行数据清洗，移除页眉、页脚、广告等噪声文本。这些无用内容会“稀释” chunk 的语义向量，降低检索精度。

### RAG生成评估

#### 生成评估目标

- 准确性（Correctness）、忠实性（Faithfulness）、表达好（Fluency）、覆盖完整（Completeness）

![1759129782466](.\resources\1759129782466.png)

- 此答案忠实剧本，语气自然，甚至有一些情感向的解释，是非常的回答

#### 生成评估方法

实际采用策略：小数据集人工评估；大数据集用大模型评估，然后人工去抽样检查

##### 机器评估指标

| 评估指标类别                                               | 核心思想                                                     | 优点                                                         | 缺点                                                         | 典型适用场景                                                 |
| :--------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **1. 基于词匹配的统计指标** （例如：BLEU, ROUGE, METEOR）  | 通过计算生成文本与参考文本之间**单词或词组（n-gram）的重合度**来打分。 | - **计算速度极快** - 定义明确，可重复性强 - 适合大规模快速筛选模型 | - **缺乏语义理解**，无法处理同义替换、语序变化 - 评估结果较为**字面化和机械** | - 机器翻译（BLEU） - 文本摘要（ROUGE） - 需要快速进行模型迭代的初期阶段 |
| **2. 基于语义相似度的指标** （例如：BERTScore, BARTScore） | 利用预训练语言模型（如BERT）生成词语的上下文向量，比较生成文本与参考文本之间的**语义相似度**。 | - **更贴近人类判断**，能理解语义相似性 - 比词匹配指标**更准确、更可靠**，是目前主流方法 | - 计算量大于词匹配指标（但仍可接受） - 其效果**依赖于所选预训练模型的质量** | - 各类文本生成任务的**核心自动评估指标** - 特别是当语义保真度比字面匹配更重要时 |
| **3. 基于问答的评估** （QA-based Evaluation）              | 不直接比较文本相似度，而是通过一个“裁判”QA模型来**检验生成文本是否包含了正确答案**。 | - 评估焦点在**信息准确性和完整性**上 - 非常适合评估**问答、摘要、RAG系统**等任务的质量 | - 效果**严重依赖于“裁判”QA模型的能力** - 构建高质量的问题集需要额外成本 | - 问答（QA）系统 - 检索增强生成（RAG）系统 - 事实性、信息性内容的评估 |

1. **共同优点**： 以上所有方法都属于**自动评估**，核心价值在于能够**低成本、高效率地处理大规模评估**，为模型迭代提供快速反馈。
2. **共同局限（非常重要）**：**无法有效识别“幻觉”**： 如果模型生成的内容逻辑通顺但事实错误（即“胡编”），只要其表述与参考文本在词法或语义上相似，这些指标仍可能给出高分。它们**缺乏真正的事实核查能力**。**依赖参考答案的质量**： 这是一种“垃圾进，垃圾出”的原则。如果作为标准答案的参考文本本身质量不高，那么任何自动评估的结果都是不可靠的。
3. **实践建议**：在项目初期或大规模实验时，可结合使用 **BERTScore** 和 **ROUGE** 等指标进行快速评估。对于问答、RAG等系统，**QA-based Eval** 是一个非常有价值的专项评估思路。**任何自动指标都不能完全替代人工评估**。在产品上线或关键决策前，必须进行**人工评估**，仔细检查生成内容的事实准确性、相关性和流畅性。

###### Rages

- 这是一款专为大模型应用设计的评估框架，其核心价值在于解决了“如何客观、自动化地评估基于检索增强生成（RAG）系统质量”这一难题。

- 传统上，评估一个 RAG 系统的效果（比如一个智能客服或文档问答机器人）需要大量人工介入，例如：

  - 人工判断答案是否正确。

  - 人工判断答案是否源自提供的文档。

  - 人工判断答案是否清晰、有用。

- 这个过程效率低下、成本高且主观性强。**Ragas 的出现，就是为了实现 RAG 系统的自动化、端到端的评估。** 它提供了一系列指标和方法，能够像运行单元测试一样，对您的 RAG 系统进行批量、可重复的评估。

---

- 核心评估维度详解：提到的三个指标 `correctness`（正确性）, `groundedness`（忠实性/依据性）, 和 `helpfulness`（帮助性）是 Ragas 评估体系的基石。它们从不同角度衡量回答的质量。

- 忠实性 - `Groundedness`这是评估的**首要且最关键的检查点**。它评估的是：**生成的答案在多大程度上严格遵循所提供的上下文（检索到的文档片段），而没有“编造”信息。**

  - **为什么要评估它？** 这是为了检测和防止模型的“幻觉”问题。一个回答可能看起来很有道理，但如果其中的关键信息没有在提供的资料中找到依据，它就是不可信的。

  - **如何评估？** Ragas 通常会使用一个经过训练的“忠实性判别模型”，该模型会分析生成的答案和检索到的上下文，判断答案中的每一个关键主张是否都能在上下文中找到支持。

  - **示例：** **上下文：** “Ragas 是一个用于评估 RAG 系统的 Python 库。”**高忠实性回答：** “Ragas 是一个用 Python 编写的库，主要用于评估检索增强生成系统。” （所有信息均源于上下文）**低忠实性回答：** “Ragas 是一个由 Google 在 2023 年发布的 Java 工具。”（“Java”和“由 Google 在 2023 年发布”都是编造的信息，上下文未提供）。

- 正确性 - `Correctness`在通过了“忠实性”检查后，流程进入“正确性”评估。它评估的是：**生成的答案与“标准答案”或事实真相的一致性。**

  - **为什么要评估它？** 答案是否忠实，只保证了它不胡编乱造，但无法保证它正确。正确性要求检索到的文档本身包含正确信息，并且模型准确地理解了这些信息。

  - **如何评估？** 这通常需要有一个“参考答案”或“黄金答案”作为基准。Ragas 会将模型生成的答案与参考答案进行对比，评估其语义上的一致性。

  - **示例：** **问题：** “Ragas 的主要用途是什么？”**参考答案：** “评估 RAG 系统。”**高正确性回答：** “它的主要功能是评估检索增强生成（RAG）应用的质量。”（语义与参考答案一致）**低正确性回答：** “它是一个用于构建 RAG 系统的框架。”（功能描述完全错误，即使它听起来合理）。
  - **`正确性`与 `忠实性`的关系**：**一个不忠实的答案，其正确性评估在很大程度上失去了意义**，因为它已经犯下了“幻觉”这个根本性错误。只有一个答案是忠实的，我们才有必要、也有可能去进行有意义的正确性评估。此时，正确性评估的**前提**是答案已经忠实于上下文。

- 帮助性 - `Helpfulness`这是流程的最终阶段，评估答案的“用户体验”。它评估的是：**答案是否直接回答了问题，并且清晰、完整、易于理解？**

  - **为什么要评估它？** 一个正确且基于事实的答案，如果冗长、跑题或难以理解，对用户来说价值也很低。例如，直接复述一大段文档而不做总结，对用户帮助不大。

  - **如何评估？** 这更侧重于答案的质量和风格，例如：是否简洁、是否结构化良好、是否避免了不必要的冗余信息、是否以用户易于理解的方式呈现。

  - **示例：** **问题：** “如何安装 Ragas？”**低帮助性回答：** “你可以用 pip 安装它。pip 是 Python 的包管理器……（此处省略200字介绍pip）……最后你可以运行 `pip install ragas`。” （包含大量无关信息）**高帮助性回答：** “使用 pip 命令即可安装：`pip install ragas`。” （直接、简洁、切中要点）。

| 指标               | 核心关注点                     | 解决的问题                   |
| :----------------- | :----------------------------- | :--------------------------- |
| **`Groundedness`** | 答案是否**源自**提供的上下文？ | 模型“幻觉”或编造信息         |
| **`Correctness`**  | 答案本身是否**正确**？         | 答案与事实或标准答案的一致性 |
| **`Helpfulness`**  | 答案是否**好用、易读**？       | 答案的清晰度、简洁性和实用性 |

**Ragas 的核心价值在于：**它将原本主观、定性的评估任务，转化为一系列可量化的指标。开发者和研究人员可以：

1. **自动化测试**：在每次对 RAG 系统（如调整检索策略、更换基础模型、修改 Prompt）进行更改后，快速运行 Ragas 评估套件，用数据判断改动是优化还是退化。
2. **精准定位问题**：如果发现评分下降，可以通过分析具体指标的得分（是 `groundedness`低了还是 `helpfulness`低了？）来快速定位瓶颈所在。例如，如果 `groundedness`得分低，说明问题出在检索或模型遵循指令上；如果 `correctness`低，可能需要优化检索资料的质量。

##### 用户反馈评分

- 上线后，通过用户的点赞或者点踩来判断
- 优点在于实用价值最高，缺点在于不太系统且偏主观
