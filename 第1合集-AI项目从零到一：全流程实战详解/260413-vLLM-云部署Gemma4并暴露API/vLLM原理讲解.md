# vLLM讲解

vLLM论文：https://arxiv.org/abs/2309.06180

vLLM 是一个专门为大语言模型（LLM）设计的**高性能推理和服务库**，它最核心的创新在于引入了 **PagedAttention** 技术，通过模仿操作系统中虚拟内存的分页管理方式，解决了显存碎片化和浪费的问题，从而**极大地提升了显存利用率和吞吐量**。我们可以使用vLLM在远程服务器部署我们的开源大模型，实现高效部署。

**LLM运行时显存利用情况**

有一句话说的好，AI时代拼的不止是模型，更是算力，如何在有限的GPU资源下，更高效的部署LLM，对整个LLM的落地有着很高的价值，下面是在 NVIDIA A100上部署130亿参数大语言模型时的显存使用情况。

- LLM的**参数**在整个部署过程中持续存在于 GPU内存中占比**65%**，一般不会变化。

- **KV Cache** 内存会**根据每个服务请求进行（释放）分配**，**占比>30%**。
- **others：**少量内存临时用于激活。

<img src="https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260401233023761.png" alt="image-20260401233018046" style="zoom:50%;" />

很明显，在GPU中 Kv Cache部分，**内存会根据每个服务请求来分配资源**，并且占GPU内存的占比也很大，所以整个vLLM的创新突破点就在于**优化GPU显存中对KV Cache的分配逻辑**。所以我们要理解 vLLM，核心就在于它如何处理 **KV Cache（键值缓存）**。

## 1. KV Cache

要了解什么是KV Cache，我们需要先了解LLM的基本架构`Transformer`，以及大模型是如何根据我们的Prompt，生成我们想要的答案的具体原理。

### 1. LLM推理原理

下面我用一个实际的Prompt：天气怎么样，来模拟LLM内部是如何根据我们的Prompt，生成我们想要的答案：晴转多云；下面是我做的一个简单的流程图，帮助大家更加清晰的掌握LLM的推理原理。

![image-20260403015012721](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260403015012800.png)

#### 1. 准备阶段：把Prompt变成“数字信号”

在被输出Transformer之前，其实还有一个隐藏的**预处理步骤**。

- **分词（Tokenization）**：模型不认识“天气”。会先把“天气怎么样”切成最小单元。例如：`["天", "气", "怎", "么", "样"]`。
- **词嵌入（Embedding）**：每个字会被转换成一个向量。这就是图中蓝色方块 $X_1 \dots X_5$ 的由来。
- **位置编码（Positional Encoding）**：为了让模型知道“天”在“气”前面，会给这些向量注入“位置信息”。

------

#### 2. Prompt 阶段

这就是图中**红框**的部分。这个阶段的目标是：**理解Prompt，并生成第一个字。**模型会为每一个输入 Token 计算其在当前上下文中的含义。以下是以 $Q_5$ 为视角，计算的完整流程：

------

![image-20260405152937506](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260405152937949.png)

##### 1. 向量转换（线性投影）

首先，模型通过三个训练好的权重矩阵 $W_Q, W_K, W_V$，将“样”字的输入向量 $X_5$ 映射到三个不同的特征空间：

- **Query（查询）**: $Q_5 = X_5 \cdot W_Q$ —— “我想找什么？”
- **Key（键）**: $K_5 = X_5 \cdot W_K$ —— “我有什么特征可以被别人查询？”
- **Value（值）**: $V_5 = X_5 \cdot W_V$ —— “如果我被选中了，我能提供什么信息？”

------

##### 2. 计算注意力原始分数（Score）

这一步是所谓的“回头看”。$Q_5$ 会分别与 $K_1, K_2, K_3, K_4, K_5$ 进行**点积（Dot Product）**运算。
$$
S_{5,j} = Q_5 \cdot K_j^T, \quad j \in \{1, 2, 3, 4, 5\}
$$

- **物理意义**：点积的结果越大，代表两个向量在空间中的方向越接近，即“语义相关性”越高。

------

##### 3. 缩放与归一化（Softmax）

为了防止点积结果过大导致梯度消失，模型先进行缩放，再通过 **Softmax** 函数将分数转化为 0 到 1 之间的概率，即**权重系数** $\alpha$。
$$
\alpha_{5,j} = \text{softmax}\left(\frac{S_{5,j}}{\sqrt{d_k}}\right) = \frac{\exp\left(\frac{Q_5 \cdot K_j^T}{\sqrt{d_k}}\right)}{\sum_{n=1}^{5} \exp\left(\frac{Q_5 \cdot K_n^T}{\sqrt{d_k}}\right)}
$$

- $d_k$ 是向量的维度（缩放因子）。
- 此时，$\alpha_{5,1} + \dots + \alpha_{5,5} = 1$。这些系数决定了模型在预测下一个词时，要把多少“注意力”放在前面的每一个字上。

------

##### 4. 加权求和（输出计算）

最后，根据刚才算出的权重 $\alpha$，对所有的 **Value** ($V$) 进行加权求和，得到 $Q_5$ 最终的上下文向量 $T_5$。
$$
T_5 = \sum_{j=1}^{5} \alpha_{5,j} V_j
$$

- **结果**：$T_5$ 不再仅仅代表“样”这个孤立的字，它现在包含了“前面是在问天气”这个完整的语义信息，蕴含了整个prompt的全部信息，所以Answer的第一个输出就选取$T_5$。

------

在 Prompt 阶段，不仅是 $Q_5$ 在看前面，**$Q_1, Q_2, Q_3, Q_4$ 都在同时进行上述计算。**

1. **$N^2$ 复杂度**：如果你输入 1000 个字，每一个字都要和 1000 个字比对，总共产生 $1000 \times 1000$ 次点积运算。
2. **KV Cache 的作用**：虽然计算量大，但所有 $K_j$ 和 $V_j$ 在这个阶段算完后，就会被存入显存。
   - 生成阶段的计算量从 $N^2$ 降级到了 $N$（线性增长），因为不需要再重新计算旧字之间的相互关系了。

------

#### 3. 生成阶段（自回归解码 Decoding）

<img src="https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260405153224023.png" alt="image-20260405153223732" style="zoom: 33%;" />

这是图中**绿框**的部分。这个阶段的目标是：**一个接一个地蹦词。**

##### 1. 从 Prompt 跨越到 Answer

在 **Prompt 阶段**结束时，最后一个输入 $X_5$（“样”）经过复杂的注意力计算，输出了一个高维向量 $T_5$。

1. **概率映射**：$T_5$ 会经过一个线性层和 **Softmax**，将其映射到整个词表（Vocabulary）上。
   - 例如：词表中“晴”的概率是 0.8，“阴”是 0.1，“雨”是 0.05...
2. **采样（Sampling）**：模型根据概率选取了“**晴**”。
3. **定义新输入**：这个“**晴**”既是输出，也瞬间变成了**下一步的新输入 $X_6$**。

------

##### 2. 预测第二个字“转”

现在模型开始处理 $X_6$（“晴”）。注意，此时显存里已经躺着预热好的 $K_1 \dots K_5$ 和 $V_1 \dots V_5$ 了。

1. **局部特征提取**：

   模型只计算当前这一个字的向量：$$Q_6, K_6, V_6$$

2. **跨时空检索（KV Cache Read）**：

   为了确定“晴”后面该接什么，模型需要回看上文。此时，$Q_6$（“晴”的查询请求）会去点积**所有的 Key**。

   - **计算公式**：$\text{Attention Score} = Q_6 \cdot [K_1, K_2, K_3, K_4, K_5, K_6]^T$
   - **关键点**：模型**直接从内存读取** $K_1 \dots K_5$，只需要对 $K_6$ 进行一次矩阵乘法。这比重新计算一遍 Prompt 省下了海量的浮点运算。

3. **信息聚合**：

   根据算出的权重，模型从所有的 Value $[V_1 \dots V_6]$ 中提取信息，算出 $T_6$。

4. **得出结论**：$T_6$ 经过分类，模型判断下一个字大概率是“**转**”（$X_7$）。

------

##### 3. 更新KV Cache 

在生成“转”（$X_7$）之前，模型会做一件重要的小事：

- **追加缓存**：把刚才算出来的 $K_6$ 和 $V_6$ 也存进显存。

------

##### 4.预测下一个输出（循环往复）

现在输入变成了 $X_7$（“转”）。

1. **计算新 QKV**：只算 $Q_7, K_7, V_7$。
2. **注意力计算**：$Q_7$ 与缓存中的 $K_1 \dots K_6$ 以及新出的 $K_7$ 进行对比。
   - 此时模型“脑子里”想的是：“天、气、怎、么、样、晴”这六个字后面跟着一个“转”，那接下来的字应该是“**多**”。
3. **缓存更新**：把 $K_7, V_7$ 存入。

------

##### 5. 递归直到终止

这个过程会一直持续（$X_8 \rightarrow$ “多”，$X_9 \rightarrow$ “云”），直到模型输出了一个特殊的 Token：**`<EOS>` (End of Sentence)**。看到它，推理进程才会正式宣告结束。

------

### 2. 两个阶段的KV Cache处理 

#### 阶段一：Prompt 阶段

- **动作**：模型一次性读入你输入的全部提示词 。
- **特点**：因为**输入的所有词都是已知的**，模型可以利用 **GPU 的并行能力**，**同时计算**所有输入词的 $K$ 和 $V$ 。
- **结果**：产生第一个输出 Token（Xn+1），**并将输入词的 KV 向量存入缓存** 。

#### 阶段二：生成阶段

- **动作**：模型一个接一个地蹦出新词 。
- **特点**：每一步只输入**上一步产生的那一个词** 。
- **效率**：此时模型只需计算这一个新词的 $Q$、$K$、$V$，然后从 KV Cache 中**读取**之前所有词的 $K$ 和 $V$ 来拼在一起计算 。
- **瓶颈**：这个阶段是**访存密集型（Memory-bound）**的，**GPU 的计算单元大多在等数据从显存搬运过来，导致利用率低下** 。

------

## 2.  KV 缓存管理的硬伤：动态需求 vs.静态分配

### 1. 核心矛盾：内存的“连续性约束”

- **现状**：传统的深度学习框架（如 PyTorch, TensorFlow）为了计算效率，要求张量（Tensor）在显存中必须占据**连续的物理空间**。
- **问题**：KV 缓存是**动态增长**的。模型每多生成一个字，缓存就变大一点。但在连续空间的要求下，系统无法像“接水管”一样随时延长内存，只能采取最原始的方法——**预分配**。

### 2. 内存碎片化：被浪费的显存空间

由于无法预测一个请求到底会生成多长，系统被迫采取了“**按最大可能容量预留**”的策略。这导致了两种碎片化：

![image-20260403000826636](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260403000826789.png)

- **内部碎片（Internal Fragmentation）**：

  - **现象**：系统为你的请求预留了 2048 个 Token 的位子，但你只问了一句“你好”，模型回了 10 个字。
  - **后果**：剩下的 2000 多个位子虽然空着，但因为已经分配给了这个请求，其他请求无法使用。

- **外部碎片（External Fragmentation）**：

  - **现象**：不同请求预分配的大小不一（如有的预留 512，有的预留 2048），导致显存空间被切得细碎。
  - **后果**：即便总空闲空间足够，但因为不连续，无法再塞进一个新的大请求。

- **直观对比**：

  <img src="https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260402235637362.png" alt="image-20260402235637302" style="zoom:50%;" />

  在传统系统中，**只有 20.4% - 38.2% 的内存是真的存了东西**，剩下的 60% 以上全是被浪费的虚位。

------

### 3. 共享障碍：孤岛式的内存分布

在大模型的高级应用场景中（如 **Beam Search 束搜索** 或 **并行采样**），模型会针对同一个 Prompt 生成多个不同的回答：

- **理想状态**：这些不同的回答在Prompt 部分的 KV 缓存是完全一样的，理论上可以**共用**一份内存。
- **现实困境**：**因为要求连续存储**，系统不得不为**每一个分支序列都完整拷贝一份一模一样的 KV 缓存**。
- **后果**：内存开销成倍增加，直接限制了模型同时处理多个任务的能力。

------

## 3. vLLM核心算法：PagedAttention

vLLM 的天才之处在于，它从操作系统的**虚拟内存（Virtual Memory）**技术中获取了灵感 。它提出了 **PagedAttention** 算法。

#### 1.OS 虚拟内存 vs vLLM

##### 1. 分页机制

**PagedAttention** 模仿了**操作系统的分页机制**，将 KV Cache 划分为固定大小的 **KV 块（KV Blocks）**，而不是分配一整块连续内存。

- **KV Block：** 每个块存储固定数量（例如 $B=4$）的 Token 的 KV 向量。
- **非连续存储：** 这些块在物理显存中**不需要连续存放**。它们可以散落在显存的任何角落。
- **逻辑连续：** 就像电脑内存一样，通过一个“页表”将物理上分散的块映射成逻辑上连续的序列。

<img src="https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260403003051844.png" alt="image-20260403003051701" style="zoom:50%;" />

**注意力得分 ($A_{ij}$):** 不是一次性算完整个序列，而是先计算查询向量 $q_i$ 与第 $j$ 个块中的所有 Key ($K_j$) 的乘积。

**加权求和 ($o_i$):** 将算出的得分与对应的 Value 块 ($V_j$) 相乘并求和：
$$
o_i = \sum_{j=1}^{\lceil i/B \rceil} V_j A_{ij}^\top
$$

##### 2.块表（Block Table）的工作机制

![image-20260403004359987](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260403004400137.png)

在模型的视角里，对话是连续的；但在 GPU 显存里，空间是零散的。

- **逻辑 KV 块 (Logical KV blocks)：** 这是模型看到的“序列”。图中 Request A 的对话被切分成了块。
  - **Block 0** 存的是前 4 个词（Four, score, and, seven）。
  - **Block 1** 存的是后 4 个词（years, ago, our, fathers）。
- **物理 KV 块 (Physical KV blocks)：** 这是显存中真实的物理位置。你可以看到 Block 7、Block 1、Block 3 是随机分布在 GPU DRAM 里的，中间夹杂着大量空闲块。

**转换过程示例：**

当模型需要读取“逻辑 Block 0”的数据时，它查阅**块表**的第一行：

> **逻辑索引 0 $\rightarrow$ 物理块号 7**

于是，系统直接去物理显存的 **Block 7** 抓取数据。

------

**块表（Block Table）的关键字段**

块表不仅仅记录了“在哪”，还记录了“状态”：

| **字段**                  | **作用**                  | **图片中的实例**                                             |
| ------------------------- | ------------------------- | ------------------------------------------------------------ |
| **Physical block number** | 物理地址指针。            | 逻辑 Block 1 映射到了物理 Block 1。                          |
| **# filled**              | 该块已填充的 Token 数量。 | 逻辑 Block 1 的数值从 $3 \rightarrow 4$，说明刚刚生成了一个新词 "fathers" 填满了这个块。 |

------

**动态转换流程：以输出 "brought" 为例**

1. **逻辑增长：** 模型生成了新词 "brought"，它属于逻辑上的 **Block 2**。
2. **物理分配：** 块管理器发现这是一个新的逻辑块，于是从显存池中随便找了一个空闲块——**物理 Block 3**，并将其分配给它。
3. **更新块表：** 在块表的第三行写入：逻辑索引 2 $\rightarrow$ 物理块号 3，并将 `# filled` 设为 1。
4. **后续预测：** 逻辑 Block 2 中剩下的 3 个位置（图中白色空格）被保留，用于存储接下来的输出 Token。

------

**这种转换带来的核心优势**

- **无感偏移：** 模型在进行注意力计算时，**依然觉得自己处理的是一个连续序列**，所有的**“跳转”和“寻址”**都由 PagedAttention 内核在后台**根据块表自动完成**。
- **显存碎片清理：** 即使显存中 Block 0、2、4、5、6、8 都是乱序空闲的，块表也能利用它们。只要还有一个空闲块，就能支持模型继续生成。
- **写时复制（Copy-on-Write）：** 虽然图中没画，但如果有一个 Request B 共享了相同的 Prompt，块表只需把 Request B 的前两个逻辑块也指向物理 Block 7 和 1，就能实现**“一份显存，多人使用**”。

------

##### 3. vLLM 处理多请求并发

![image-20260403005951330](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260403005951486.png)

**全局视图：多序列并发管理**

图片右侧引入了 **Request B**，这展示了 vLLM 在处理真实服务器负载时的强大之处：

- **物理池共享：** 你可以看到，物理显存（Physical KV blocks）是一个统一的池子。
  - Request A 占用了块 7, 1, 3。
  - Request B 占用了块 5, 2。
- **非连续性：** Request B 的逻辑块 0 映射到物理块 5，逻辑块 1 映射到物理块 2。这些物理块在显存中完全不连续，且与 Request A 的块交织在一起。
- **内存浪费极低：** 观察图中所有的块，你会发现只有每个 Request 的**最后一个块**（Request A 的块 3，Request B 的块 2）存在空白槽位。这意味着内存浪费被限制在 $Batch\_Size \times Block\_Size$ 之内，利用率极高。

## 4. vLLM在复杂任务中实现极致的显存共享

### 1. 并行采样（Parallel Sampling）：写时复制（Copy-on-Write）

**场景：** 针对同一个输入提示（Prompt），模型生成多个不同的采样结果（如代码助手的多个候选建议）。

![image-20260403010906371](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260403010906526.png)

- **核心机制：引用计数 (Reference Counting)**
  - 因为所有输出都**共享同一个 Prompt**，它们的逻辑块**最初都指向同一个物理块**。
  - 每个物理块都有一个“引用计数”，记录有多少个逻辑块在使用它。
- **写时复制 (CoW)：**
  - **触发 CoW：** 当 A1 准备写入 "fathers" 时，系统检查到物理 Block 1 的引用计数为 2。
  - **物理拷贝：** 块管理器在显存中找了一个空闲的 **物理 Block 3**，并将物理 Block 1 原有的数据（"years", "ago", "our"）完整地**复制**过去。
  - **修改引用计数：** 原物理 Block 1 的引用计数从 **2 降到了 1**。
  - **独立写入：**
    - **Sample A1** 的逻辑 Block 1 现在指向新的 **物理 Block 3**，并将 "fathers" 写入其中。
    - **Sample A2** 的逻辑 Block 1 仍然指向原来的 **物理 Block 1**，并将 "mothers" 写入其中。

------

### 2. 波束搜索（Beam Search）：动态树状共享

![image-20260403011558538](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260403011558703.png)

**场景：** 机器翻译等任务中，每一步保留得分最高的 $K$ 个候选序列。

- **复杂的共享模式：** 波束搜索的共享不仅发生在**初始 Prompt，还发生在生成过程中的共同分支**。
- **动态变化：** 随着解码进行，某些“波束”被淘汰（逻辑块被释放，引用计数减 1），某些“波束”分叉出新序列。
- **优势：**
  - **减少拷贝：** 传统的系统在波束分叉时需要拷贝大量的 KV Cache。
  - **自动回收：** 当引用计数归零时，vLLM 会立即释放物理块（如文中提到的 Block 2, 4, 5, 8），并分配给新的候选序列。

------

### 3. 共享前缀：类比共享库

![image-20260405230250300](https://cdn.jsdelivr.net/gh/SkyMoem/Bed-Images/img/20260405230251046.png)

**场景：** 许多请求拥有相同的“系统提示词”（System Prompt）或指令（Instruction）。

- **实现方式：** vLLM 可以预先计**算并缓存这些长前缀的 KV Cache**，将其**存储在固定的物理块中**。
- **效率提升：** 新请求进来时，由于前缀已缓存，**系统只需计算用户新输入的部分**，**极大降低了首字延迟（TTFT）**。

------

### 4. 混合解码（Hybrid Decoding）：抽象的力量

vLLM 最强大的地方在于它构建了一个**公共映射层**。

- **屏蔽复杂性：** 无论底层是并行采样、波束搜索还是共享前缀，LLM 模型本身和执行内核（Kernel）看到的都是一串简单的物理块 ID 列表。
- **统一批处理：** 这种抽象允许 **vLLM 将完全不同类型的请求（有的在采样，有的在波束搜索）打包进同一个 Batch 进行处理**，从而最大化 GPU 的吞吐量。

| **场景**     | **共享对象**   | **核心机制**              | **收益**                     |
| ------------ | -------------- | ------------------------- | ---------------------------- |
| **并行采样** | 初始 Prompt 块 | 引用计数 + 写时复制 (CoW) | 节省多副本重复内存。         |
| **波束搜索** | 历史生成块     | 动态树状路径映射          | 避免频繁的大规模内存拷贝。   |
| **共享前缀** | 系统/任务描述  | 物理块持久化缓存          | 降低首字延迟，省去冗余计算。 |

## 5.vLLM的其他机制

vLLM 通过 **FCFS（先来后到）** 调度策略，在显存溢出时优先“保老弃新”，利用**全有或全无**的原则批量抢占请求，并通过**重计算或交换**机制灵活恢复任务；在分布式环境下，它采用**中央调度器**统一管理物理块映射，配合**张量并行**将 KV 缓存分片存储在各 GPU 中，实现了存取时无需跨卡通信、计算时高效同步的“逻辑统一、物理分片”架构，从而在保障系统稳定性的同时极大**提升了超大规模模型的并发吞吐能力**。这块我们了解一下就可以，不是重点内容；

## 7. 为什么选择vLLM？

### 1. 协议标准化：OpenAI 兼容性

vLLM 最受开发者欢迎的特性之一是它原生支持 **OpenAI API 协议**。这意味着你不需要为 vLLM 编写专门的 SDK。

- **无缝切换**：如果你原本的代码是调用 GPT-4，你只需要更改 `base_url`（指向你的 vLLM 服务器地址）和 `api_key`（自定义或留空），现有的 Python/JS OpenAI SDK 就能直接驱动你本地部署的开源模型。
- **集成方便**：主流的 LLM 开发框架（如 **LangChain**、**LlamaIndex**）都有成熟的 vLLM 接口。

------

### 2. 生产级的吞吐量：PagedAttention

在云端服务中，“成本”和“延迟”是核心指标。vLLM 的核心黑科技是 **PagedAttention** 算法。

- **显存利用率**：传统引擎在处理长文本时会预留大量连续显存，导致浪费。vLLM 像操作系统管理内存分页一样管理显存（KV Cache），利用率接近 **100%**。
- **高并发**：在同样的显存条件下，vLLM 能处理的并发请求数通常是 Ollama 或原生 HuggingFace 部署的 **5-10 倍**。

------

### 3. 企业级特性：分布式与动态扩容

相比于 Ollama 和 LM Studio 这种单机运行的工具，vLLM 专为集群设计：

- **多卡张量并行 (TP)**：当一个模型单张显卡塞不下时，vLLM 可以轻松实现跨多张显卡的分布式推理。
- **连续批处理 (Continuous Batching)**：它不需要等待一整批请求结束才开始下一批，而是随到随处理，极大地降低了首字延迟（TTFT）。

------
