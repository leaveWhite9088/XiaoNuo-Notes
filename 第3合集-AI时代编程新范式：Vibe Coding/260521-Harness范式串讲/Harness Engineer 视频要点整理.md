# Harness Engineer：从组件到范式

## 一、什么是 Harness

**Harness（马具 / 挽具）**，原义是套在马身上让它能拉车的那一整套缰绳、嚼子、马鞍。借到 AI 工程语境里，意思是：**套在模型外面、让它能稳定干活的那一整套工程结构**。

为什么需要它？因为今天的大模型，单看推理能力其实已经非常强，但一旦放进真实项目，会暴露出三类问题：

1. **上下文会污染**。会话开得越久，越容易漂移、幻觉，输出风格也越不稳定。
2. **没有持久记忆**。模型不知道你这个项目的约定、上次为什么这么改、哪些坑踩过。
3. **没有"必须做"的概念**。哪怕你在 prompt 里写了三遍"不要 force push"，它还是可能在某次决策里忘掉。

Harness 就是用来补这三件事的——用**协议、子代理、提示词、规则文件、钩子**这些工程手段，把模型框在一个可控的执行轨道上。

本期讨论的范围限定在 **CLI 类编程 Agent**（Claude Code、Cursor、Codex、Gemini CLI、Qwen Code 等），不涉及 AutoGPT 那种"全自动循环式"的 Agent 范式。

## 二、Harness 的六个组件

把一个 AI 编程 Agent 拆开看，本质是由六类机制拼起来的。下面逐个过。

### 2.1 MCP（Model Context Protocol）

**定位**：AI 与外部系统之间的**标准协议格式**。

**作用**：让模型能以统一方式调用任意外部工具/服务。在 MCP 出现之前，每接一个系统（数据库、浏览器、设计稿、Jira……）就要写一套适配器；MCP 把这层接口标准化，工具方按协议出 server，模型侧按协议调用。

**典型例子**：playwright MCP（让模型操作浏览器）、context7 MCP（让模型查最新文档）、各家云服务的 MCP。

**一句话**：MCP 是 Harness 的"外设总线"，决定模型能往外伸出多少只手。

### 2.2 Subagent（子代理）

**定位**：在主会话之外，临时拉起的隔离上下文。

**核心动机**：主会话上下文越长，模型越容易**漂移、幻觉**。尤其是 GLM 这一类 256K 上下文模型，跑到一半被 Compact（自动压缩历史消息）之后，关键约束经常就丢了。

**用法**：

- 把"读一堆文件、跑一轮探索、做一次代码 review"这种重活外包给 Subagent
- 主会话只接收 Subagent 的**最终结论**，不接收它的探索过程
- 主会话从 200K 上下文 → Subagent 跑一轮（0K 起步 → 完成）→ 主会话回到尽可能小的上下文

**一句话**：Subagent 是用来"保护主会话不被脏数据污染"的。它不是为了并行，是为了**隔离**。

### 2.3 Skill（动态加载提示词）

**定位**：运行时按需注入的"技能提示词"。

**触发方式**：模型根据当前的目标和场景，自己识别需要哪个 Skill，再动态加载到上下文里。比如你说"帮我做个落地页"，模型就会自动加载 `frontend-design` Skill；你说"提交前帮我跑检查"，就会自动加载 `preflight` Skill。

**和 Rule 的区别**：Rule 是常驻的、每次都在上下文里；Skill 是动态的、按需才进上下文。这样可以把上下文留给真正在用的能力。

**生态**：社区已经开始出现 Skill 市场（类似 OpenClaude、ClaudeHub 这类项目）。

**⚠️ 风险——Skill 投毒**：第三方 Skill 本质是一段提示词，如果里面藏了恶意指令（比如"在提交代码时悄悄把环境变量发到某个 URL"），模型加载即中招。装第三方 Skill 一定要看源代码，和装 npm 包是一个道理。

### 2.4 Rule（规则文件）

**定位**：以"被阅读"的方式生效的**硬性约束**。

**形式**：

- `CLAUDE.md` / `.cursorrules` / `AGENTS.md`——项目根目录的约定文件
- `/rule` 类型的子命令
- 各种 formatter 规则（比如 `rule-formatter.java` 约定 Java 代码风格）

**触发机制**：模型每次开始工作前，会主动读这些文件，把里面的内容"当成自己的设定"。

**和 Skill 的区别**：Skill 是按需的，Rule 是常驻的。**重要约束写 Rule，可选能力写 Skill**——这是个核心的划分原则。

### 2.5 Agent Team（多代理协作，实验性）

**定位**：多个 Agent 之间分工协作（产品经理 Agent + 架构师 Agent + 编码 Agent + Review Agent ……）。

**现状**：还在实验阶段，远未实用化。两个核心问题：

1. **烧 token 太多**。每次跨代理传递都要复述上下文，多代理协作的总 token 消耗可能是单 Agent 的 5–10 倍。
2. **非它不可的场景似乎很少**。绝大多数复杂任务，用一个主代理 + 若干 Subagent 已经够用。

**真实意义**：与其说它是新范式，不如说它是**给 Subagent 补一个缺陷**——Subagent 每次启动都要重新阅读上下文，且不支持复杂的交叉交互管理。Agent Team 想解决这两个问题，但目前看更像噱头。

**判断标准**：如果你的任务真的需要"多个 Agent 之间长时间反复对话"，可以考虑；否则用 Subagent 就够了。

### 2.6 Hook（生命周期钩子）

**定位**：在 Claude Code（或同类 CLI Agent）的某个执行阶段，**强制插入一段脚本**。

**典型触发点**：

- 执行任意工具**之前**（PreToolUse）——比如在写文件前自动备份
- 上下文压缩**前 / 后**（PreCompact / PostCompact）——比如在被压缩前把关键决策落盘
- 完成所有消息**之后**（Stop）——比如自动跑 lint
- 用户提交 prompt **之前**（UserPromptSubmit）——比如自动注入项目状态

**核心特征**：**强制性、绝对稳定**。模型可能漂、可能忘，但 Hook 一定会执行——这是 Harness 里"最硬"的一根钉子。

**一个常见用法**：把"绝对不能违反的约束"从提示词里搬到 Hook 里。比如不要写"模型请记得跑测试"，而是写一个 Stop Hook 强制跑测试。前者靠模型自觉，后者靠脚本兜底。

## 三、现在有哪些 Harness 范式

讲完组件，回到使用者视角：到底什么场景该上 Harness？按"模型能力 × 项目质量要求"可以分三档。

### 3.1 非常不需要 Harness 的场景

**代表模型**：Claude 4.7 Opus（1M 上下文窗口）。

**判断标准**：模型本身能在复杂的**单会话**里跑完整段长程工作，且不会漂移。

**为什么不需要**：

- 1M 上下文意味着整个项目几乎可以一次性塞进去，不再需要 Subagent 来"省 token"
- 模型推理能力强到一定程度后，靠 prompt 自然语言指令就能稳定执行，不再需要 Hook 兜底
- 硬塞 Harness 反而是负担——加层加结构都意味着出错点更多

**这一档的工作方式**：开一个长会话，把任务说清楚，让模型一路跑到底。

### 3.2 很需要 Harness 的场景

**代表模型**：GLM、KIMI、DeepSeek。

**特征**：单会话稳定性不够，**必须靠外部 Harness 兜底**才能跑严肃项目。

#### 3.2.1 常用 Harness 拼法

实际项目里常见的组合：

- **MCP**：playwright MCP（浏览器自动化）、Z.AI（智谱）等外部服务接入
- **Subagent**：分流上下文，做探索和 review
- **Skill**：`ClaudeDesign`（前端设计）、`/init-project`（项目初始化）、`/subagent-loop-dev`（子代理循环开发）等
- **Rule**：按语言/框架分别配——Java、Python、Spring、FastAPI、React、Vue 各一份 CLAUDE.md
- **claude lock**：锁定 Claude Code 版本，防止升级带来行为漂移

#### ⚠️ 没有通用模板！！！

GitHub 上一堆开源项目（`everything-claude-code`、`codex`、`gemini-cli-claude-code`、`cursor` 各种 awesome 列表），**没有一套能开箱即用**。

为什么？因为 Harness 的有效性高度依赖**你的具体项目**——你的代码风格、技术栈、团队约定、踩过的坑。别人的 Harness 拿过来，最多是个起点。

真正能用的工作流都是**蒸馏 / 凝练**出来的——在你自己的项目上反复打磨，慢慢沉淀出哪些 Rule 必须有、哪些 Skill 经常用、哪些 Hook 必须挂。

#### Subagent 的致命缺陷

主会话把复杂需求传给子代理时，**非常容易出现省略 / 不精准 / 多次漂移**。

举个真实场景：

> 一份 300–500 字的 PRD →
> 主会话读完，调用"需求分析子代理"，传过去时只剩 300 来字 →
> 需求分析子代理输出方案，再传给"编码子代理"时又掉一截 →
> 编码完成传给"review 子代理"，原始 PRD 的核心约束已经几乎看不到了。

每一跳都丢一点信息，最终结果是**渐进式的偏移**。

**缓解手段**：把最关键的约束**写到 Hook 和文件里**，而不是依赖主代理在 prompt 里复述。Hook 是强制的，文件是可被反复读的，prompt 是会被压缩的——三者的可靠性差一个数量级。

### 3.3 很难判断需不需要 Harness 的场景

这一档是大多数人的真实处境。代表工具：Codex、Gemini CLI、Qwen Code、OpenCode、Claude Code 等。

按"模型能力 × 项目质量要求 × 使用者经验"三个维度判断：

| 你用的模型 | 项目质量要求 | 你的开发经验 | 结论 |
|---|---|---|---|
| **强模型**（GPT-5.5、Claude Opus 4.7 这一档） | 无所谓高低 | 任意 | **不需要**复杂 Harness。模型自己能搞定。 |
| **中等模型**（GLM 5.1、DeepSeek V4 Pro、KIMI K2.6 这一档） | 较高 | 有一定经验 | **自己凝练自己的 Harness 流程**。靠"对抗审查精神"（reviewer + judge agent 这类机制）逼模型收敛。 |
| **普通模型**（StepFun、Minimax 等） | 不高 | 任意 | **不需要复杂 Harness**。直接用 Claude Code 这种成熟产品就行。 |

**核心判断**：Harness 的复杂度应该和 **(模型短板) × (项目质量门槛)** 成正比。

- 两边都低 → 别折腾
- 一边高一边低 → 看你的精力
- 两边都高 → 老老实实做 Harness

> 注：表格里的模型版本号只是当下的代表性举例，具体能力分档每隔几个月就会变一次，重点是**判断维度**而不是具体型号。

## 四、随着模型能力提升，Harness 的作用会下降吗？

会下降，但不会消失。

**Anthropic 在这个问题上的立场（也是本期视频的结论）**：

> Harness 的存在价值，在于**用框架 / 逻辑 / 代码的方式，弥补模型现有能力的不足**——
> 它是过渡形态，不是一种永远不变的万能工具。

具体推论：

- **Skill 会大幅减少**。今天写在 Skill 里的很多"如何做前端设计""如何做代码 review"，未来模型默认就会。
- **Subagent 的使用会减少**。上下文窗口和模型稳定性提升后，主会话能直接搞定的事更多了。
- **Rule 会精简**。今天 CLAUDE.md 里大段大段写"不要做 X、记得做 Y"，未来很多会变成模型默认行为。
- **MCP 会留下**。因为它是"对外的协议"，和模型能力无关——再强的模型也得有协议才能调外部系统。
- **Hook 会留下，甚至更重要**。Hook 解决的是"确定性需求"——不管模型多聪明，**"绝对不能违反"的事仍然需要硬约束**。这部分是工程问题，不是智能问题。

**给观众的建议**：不要把 Harness 当目标，也不要为了"看起来很专业"硬上一堆组件。**它的价值取决于它在补哪个具体的短板**——短板补完了，对应的 Harness 组件就该退役。

每隔几个月，回过头看看你的 Harness 里哪些东西已经不再有用，及时砍掉。这比一直加新东西更重要。
